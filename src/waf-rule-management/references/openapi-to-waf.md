# OpenAPI to WAF Rules — Positive Security Steering

Generate **allowlist rules from OpenAPI 3.x specifications** that validate requests
BEFORE CRS rules execute. The API contract defines what is allowed; CRS catches everything else.

## Quick Use

```bash
python scripts/openapi_to_rules.py openapi.yaml -o before-crs-rules.conf
python scripts/validate_rule.py before-crs-rules.conf
```

## Inclusion Order (Critical)

Generated rules **must** be loaded before CRS. The order:

```
1. OpenAPI-derived positive-security rules    ← generated by this skill
2. REQUEST-900-EXCLUSION-RULES-BEFORE-CRS     ← runtime exclusions
3. CRS rules (910-959)                        ← OWASP Core Rule Set
4. RESPONSE-999-EXCLUSION-RULES-AFTER-CRS     ← configure-time exclusions
```

If rules are placed after CRS, they cannot pre-empt CRS detections.

## What to Generate

The agent (or script) should produce rules for each of these categories. Use the
OpenAPI spec's `paths`, `parameters`, `requestBody`, `security`, and `components`
sections as the source of truth. Categories are listed in recommended implementation order.

### 1. Allowed Paths and Methods

For each path in the spec, emit a chained rule that matches the path and rejects
any method not listed.

```conf
# Allow only GET and POST for /api/v1/users/{id}
SecRule REQUEST_URI "@rx ^/api/v1/users/[^/]+(/)?$" \
    "id:200001,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule REQUEST_METHOD "!@rx ^(GET|POST)$" \
        "deny,\
        status:405,\
        log,\
        msg:'Method not allowed for /api/v1/users/{id}',\
        logdata:'%{MATCHED_VAR}',\
        tag:'openapi/method-violation',\
        tag:'positive-security',\
        severity:'WARNING'"
```

**Key points**:
- Convert OpenAPI `{param}` path segments to `[^/]+` regex.
- `@rx ^/...(/)?$` anchored pattern avoids partial prefix matches.
- The first rule in the chain carries `id`, `chain`, `nolog`, `pass`.
- The second rule carries the disruptive action (`deny`) and all metadata.
- Use `tag:'openapi/method-violation'` for filtering and exclusion.

### 2. Required and Allowed Query Parameters

For query parameters defined in the spec, validate:
- **Required parameters are present** (use `&ARGS:name @eq 0` to test absence).
- **Unexpected parameters are flagged** (optional — strict mode only).

```conf
# Require 'page' query parameter on GET /api/v1/items
SecRule REQUEST_URI "@rx ^/api/v1/items(/)?$" \
    "id:200010,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule &ARGS:page "@eq 0" \
        "deny,\
        status:400,\
        log,\
        msg:'Missing required parameter: page',\
        tag:'openapi/missing-param',\
        tag:'positive-security',\
        severity:'WARNING'"
```

For `enum` constraints, emit a rule that rejects values not in the allowed set:

```conf
# Enum validation: sort must be 'asc' or 'desc'
SecRule REQUEST_URI "@rx ^/api/v1/items(/)?$" \
    "id:200011,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule ARGS:sort "!@rx ^(asc|desc)$" \
        "deny,\
        status:400,\
        log,\
        msg:'Invalid enum value for sort parameter',\
        logdata:'%{MATCHED_VAR}',\
        tag:'openapi/param-enum-violation',\
        tag:'positive-security',\
        severity:'WARNING'"
```

### 3. Parameter Type and Pattern Validation

Use the OpenAPI `type`, `format`, `pattern`, `minLength`, `maxLength` to emit
type-checking rules on query, path, and header parameters.

| OpenAPI Type/Format | ModSec Pattern | Example |
|---------------------|----------------|---------|
| `type: integer` | `@rx ^[0-9]+$` | IDs, page numbers |
| `type: string, format: uuid` | `@rx ^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$` | Resource IDs |
| `type: string, format: email` | `@rx ^[^@]+@[^@]+\.[^@]+$` | Email fields |
| `type: string, format: date` | `@rx ^[0-9]{4}-[0-9]{2}-[0-9]{2}$` | Date params |
| `pattern: "^[a-zA-Z0-9_]+$"` | Use pattern directly with `@rx` | Custom tokens |
| `maxLength: 100` | `@rx ^.{101,}$` to reject | Length limits |

```conf
# Type check: 'id' must be integer
SecRule REQUEST_URI "@rx ^/api/v1/items/[^/]+(/)?$" \
    "id:200020,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule ARGS:id "!@rx ^[0-9]+$" \
        "deny,\
        status:400,\
        log,\
        msg:'Parameter id must be an integer',\
        logdata:'%{MATCHED_VAR}',\
        tag:'openapi/type-violation',\
        tag:'positive-security',\
        severity:'WARNING'"
```

### 4. Expected Content-Type Headers

For endpoints with `requestBody`, the OpenAPI spec lists accepted content types
(e.g. `application/json`, `multipart/form-data`). Reject requests that do not match.

```conf
# Content-Type enforcement: POST /api/v1/items must be application/json
SecRule REQUEST_URI "@rx ^/api/v1/items(/)?$" \
    "id:200030,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule REQUEST_METHOD "@streq POST" "chain"
        SecRule REQUEST_HEADERS:Content-Type "!@rx ^(application/json)" \
            "deny,\
            status:415,\
            log,\
            msg:'Unsupported Content-Type for POST /api/v1/items',\
            logdata:'%{MATCHED_VAR}',\
            tag:'openapi/content-type-violation',\
            tag:'positive-security',\
            severity:'WARNING'"
```

**Key points**:
- Use a three-deep chain: path → method → Content-Type.
- Match the start `^application/json` to allow charset suffixes (`application/json; charset=utf-8`).
- If multiple content types are allowed, join with `|` alternation.
- LLM/user should review and amend when the API accepts content types not in the spec
  (e.g. browser form submissions, file uploads).

### 5. Expected API Key and Auth Headers

If the OpenAPI spec declares `securitySchemes`, generate rules that verify the expected
auth mechanism is present. **Do not validate the token value** — just confirm the
header/parameter/cookie exists where the spec requires it.

Common OpenAPI security schemes and their WAF checks:

| Scheme Type | OpenAPI `in` | WAF Variable | Check |
|-------------|-------------|--------------|-------|
| `apiKey` | `header` | `REQUEST_HEADERS:X-API-Key` | Header present |
| `apiKey` | `query` | `ARGS:api_key` | Query param present |
| `apiKey` | `cookie` | `REQUEST_COOKIES:session` | Cookie present |
| `http` (bearer) | `header` | `REQUEST_HEADERS:Authorization` | `@rx ^Bearer\s` |
| `http` (basic) | `header` | `REQUEST_HEADERS:Authorization` | `@rx ^Basic\s` |

```conf
# Require Authorization: Bearer header on protected endpoint
SecRule REQUEST_URI "@rx ^/api/v1/admin" \
    "id:200040,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule &REQUEST_HEADERS:Authorization "@eq 0" \
        "deny,\
        status:401,\
        log,\
        msg:'Missing Authorization header for protected endpoint',\
        tag:'openapi/auth-missing',\
        tag:'positive-security',\
        severity:'CRITICAL'"
```

```conf
# Require X-API-Key header (apiKey scheme)
SecRule REQUEST_URI "@rx ^/api/v1/" \
    "id:200041,\
    phase:1,\
    chain,\
    nolog,\
    pass"
    SecRule &REQUEST_HEADERS:X-API-Key "@eq 0" \
        "deny,\
        status:401,\
        log,\
        msg:'Missing X-API-Key header',\
        tag:'openapi/auth-missing',\
        tag:'positive-security',\
        severity:'CRITICAL'"
```

**Key points**:
- Check **presence** with `&VARIABLE @eq 0` (count-based), not the token value itself.
- For bearer auth, optionally validate the `Bearer ` prefix format.
- Security scheme requirements may vary per-path — respect the per-operation `security`
  overrides in the spec, not just the global `security` block.
- **LLM/user must review**: auth requirements frequently differ between what the spec
  declares and what the real deployment enforces (e.g. gateway-issued tokens, mTLS).

### 6. Request Body JSON Structure (Optional)

For `application/json` request bodies, basic structural checks are possible using
`@rx` on `REQUEST_BODY`. Only generate these for high-value endpoints; deep JSON
validation is better handled by the application layer.

```conf
# Reject empty body on POST /api/v1/items (expects JSON object)
SecRule REQUEST_URI "@rx ^/api/v1/items(/)?$" \
    "id:200050,\
    phase:2,\
    chain,\
    nolog,\
    pass"
    SecRule REQUEST_METHOD "@streq POST" "chain"
        SecRule REQUEST_BODY "@rx ^$" \
            "deny,\
            status:400,\
            log,\
            msg:'Empty request body for POST /api/v1/items',\
            tag:'openapi/body-empty',\
            tag:'positive-security',\
            severity:'WARNING'"
```

Body rules use **phase:2** (request body access required). Keep these simple — CRS
already handles injection patterns. The goal is shape validation, not content scanning.

## Tagging Convention

All generated rules should use consistent tags for filtering and exclusion:

| Tag | Purpose |
|-----|---------|
| `positive-security` | Marks all OpenAPI-derived rules |
| `openapi/method-violation` | Method not allowed for path |
| `openapi/missing-param` | Required parameter absent |
| `openapi/param-enum-violation` | Parameter value outside enum set |
| `openapi/type-violation` | Parameter type/format mismatch |
| `openapi/content-type-violation` | Unexpected Content-Type |
| `openapi/auth-missing` | Required auth header/key absent |
| `openapi/body-empty` | Missing or empty request body |

These tags allow targeted exclusions when a legitimate client diverges from the spec:

```conf
# Exclude positive-security method checks for monitoring agent
SecRule REQUEST_HEADERS:User-Agent "@contains MonitoringAgent" \
    "id:100200,\
    phase:1,\
    pass,\
    nolog,\
    ctl:ruleRemoveByTag=openapi/method-violation"
```

## Rule ID Range

Use **200000–299999** for OpenAPI-derived positive-security rules (distinct from custom
detection rules at 100000–199999 and CRS rules at 9xxxxx).

The script default `--base-id 200000` follows this convention.

## Chaining Best Practices

OpenAPI rules rely heavily on chained rules. Follow these patterns:

1. **First rule in chain**: path match (`@rx` or `@beginsWith`), carries `id`, `chain`, `nolog`, `pass`.
2. **Middle rules** (if multi-condition): additional checks, carry `chain` only.
3. **Last rule in chain**: the disruptive action (`deny`/`pass`), `log`, `msg`, `tag`, `severity`.

Why chain instead of standalone rules:
- Chains scope the negative check to a specific path — a standalone `!@rx ^(GET|POST)$` on `REQUEST_METHOD` would fire globally.
- Chains prevent false positives on unrelated endpoints.
- Chains keep the rule set maintainable — one chain per endpoint/check.

**Anti-pattern**: Do not put `deny` on the first rule of a chain. Only the **last** rule
in a successful chain triggers the disruptive action.

## Workflow: From Spec to Deployed Rules

1. **Generate**: `python scripts/openapi_to_rules.py openapi.yaml -o before-crs-rules.conf`
2. **Review**: Check generated rules against the actual API behavior. Amend:
   - Add content types used in practice but missing from the spec.
   - Adjust auth header checks if the deployment uses a gateway/CDN for auth.
   - Remove required-param rules for params that are optional in practice.
3. **Validate**: `python scripts/validate_rule.py before-crs-rules.conf`
4. **Lint regex**: `python scripts/lint_regex.py before-crs-rules.conf -v`
5. **Test**: Write go-ftw tests covering both allowed and blocked requests:
   ```bash
   go-ftw run --cloud --config assets/docker/.ftw.yaml -d tests/api-positive-security/
   ```
6. **Deploy**: Include before CRS. Start in `detect` mode (`--mode detect`) to log
   without blocking, then promote to `enforce` after verifying no false positives.
7. **Maintain**: When the API spec changes, regenerate and diff. Review deltas before deploying.

## What the Agent Should Do (LLM Steering)

When an agent/LLM is asked to create positive-security rules from an OpenAPI spec:

1. **Parse** the spec and identify all paths, methods, parameters, request bodies, and security schemes.
2. **Generate rules** in the order listed above (paths/methods first, auth last).
3. **Use chained rules** — never use standalone negative matches on global variables.
4. **Tag consistently** using the convention above.
5. **Flag items for human review**:
   - Auth header rules — always note these may need adjustment.
   - Content-Type rules — always note the app may accept types not in the spec.
   - Required parameter rules — note that optional-in-practice params may cause FPs.
6. **Include both positive and negative go-ftw tests** for each generated rule.
7. **Suggest detect mode first** before enforce, especially for production APIs.

**Anti-patterns to avoid**:
- Generating rules without `tag:'positive-security'`.
- Using broad path prefixes that overlap with other endpoints.
- Validating auth token values (only check presence/format).
- Putting body-checking rules in phase:1 (body is only available in phase:2).
- Generating rules for paths/params inherited from `$ref` without resolving them first.

## Best Practices

- Generate rules in **detect mode first** (`pass,log`) before switching to enforce (`deny,status:403`), especially for production APIs.
- Tag every generated rule with `tag:'positive-security'` for easy filtering and exclusion.
- Include both positive tests (valid API call passes) and negative tests (invalid call blocked) for each generated rule.
- Resolve all `$ref` references in the OpenAPI spec before generating rules — unresolved refs create incomplete rules.
- Place generated rules **before** CRS include — they should allowlist valid API traffic before CRS applies generic detection.
- Review generated rules for overly broad path prefixes that might overlap with other endpoints.
- Regenerate rules whenever the API spec changes — stale rules create false positives or coverage gaps.
