# First-Responder Risk Runbook (Standardized)

Use this runbook for fast triage of potential WAF exposure from newly announced, emerging, or not-yet-fully-documented vulnerabilities.

## Scope

- Standardized workflow regardless of engine (ModSecurity or Coraza).
- Supports **concurrent investigations** — each incident gets an isolated workspace.
- Test the same vulnerability payload set across latest and prior CRS versions.
- Capture detailed request/response + WAF evidence.
- Produce a consolidated results table for first responders and decision makers.

## Key Principles

- **go-ftw is baseline/regression tooling** (repeatability, CI guardrails).
- **First response for unknown risk is probe-driven triage** using CRS+Albedo, logs, and controlled payload variants.
- **Each incident is isolated** — separate rules, tests, probes, and reports. Multiple investigations can run in parallel without interference.
- **Content-type matters** — CRS rules behave differently based on how payloads are delivered. Always test across JSON, multipart, and urlencoded.

> **Container runtime**: All `docker compose` / `docker logs` commands in this runbook work with `finch compose` / `finch logs` as a drop-in replacement.

## Incident Workspace Layout

```
incidents/
  CVE-2025-55182/           # one directory per incident
    rules.conf              # virtual-patch rules for this incident
    tests/                  # go-ftw regression tests for these rules
      100000.yaml
    REPORT.md               # structured incident report
  RISK-2026-003/
    rules.conf
    tests/
    REPORT.md
assets/docker/
  .ftw.yaml                 # shared go-ftw config (pass via --config)
  docker-compose.yaml       # shared test environment
  custom-rules.conf         # assembled active rules (generated by assemble_rules.sh)
```

Rule IDs are allocated in 100-ID blocks (100000–100099, 100100–100199, etc.) to avoid collisions between concurrent incidents. The scaffold script prints the next available block.

## Standard Playbook

### 1) Research the risk and scaffold workspace

Create an incident workspace and begin researching. The workspace is where all research, probes, rules, tests, and findings live.

**Scaffold the workspace:**

```bash
bash scripts/new_incident.sh <INCIDENT_ID>
```

This creates `incidents/<INCIDENT_ID>/` with `rules.conf`, `REPORT.md`, and `tests/` directory. The script prints the next available rule ID range — use this range for any custom rules.

**Populate the files following these conventions:**

#### `REPORT.md` — Incident Report

Write a structured report as you research. Include these sections:

```markdown
# Incident Report: <INCIDENT_ID>

## Risk ID / Advisory
- **ID**: <INCIDENT_ID>
- **Description**: <short description>
- **Advisory URL**: <link>
- **CVSS**: <score>
- **Date**: <YYYY-MM-DD>

## Hypothesis
<What is the vulnerability? How does the attack work? What should the WAF detect?>

## Probe Matrix
| ID | Description | Vector | Content-Type | Expected |
|----|-------------|--------|--------------|----------|
| P1 | <attack desc> | <payload> | application/json | 403 |
| P1-MP | <same, multipart> | <payload> | multipart/form-data | 403 |
| C1 | Normal GET | benign | — | 200 |

## Results: CRS Baseline (no custom rules)
| Probe | HTTP | Verdict | Rule IDs | Notes |
|-------|------|---------|----------|-------|

## Results: With Virtual Patches
| Probe | Before | After | Custom Rule |
|-------|--------|-------|-------------|

## Sandbox Comparison
| Payload | Local | Sandbox | Match? |
|---------|-------|---------|--------|

## Gaps Identified
1. <gap>

## Custom Rules Written
| Rule ID | Description | File |
|---------|-------------|------|

## Follow-Up
| Action | Owner | Due | Status |
|--------|-------|-----|--------|

## Environment
| Engine | Version | CRS Version | PL | Inbound Threshold | Image Tag |
|--------|---------|-------------|----|-------------------|-----------|
| ModSecurity | v3.x | v4.x | 1 | 5 | owasp/modsecurity-crs:nginx |
| Coraza | vX.x | v4.x | 1 | 5 | ghcr.io/coreruleset/coraza-crs:caddy-alpine |
```

#### `rules.conf` — Virtual-Patch Rules

Write ModSecurity SecRules using the allocated ID range. Convention:

```apache
# Virtual Patch: <INCIDENT_ID>
# Rule ID range: <BASE>–<BASE+99>
# Status: TEMPORARY — remove after upstream CRS adds coverage

SecRule VARIABLES "@OPERATOR pattern" \
    "id:<RULE_ID>,\
    phase:2,\
    deny,\
    status:403,\
    log,\
    t:none,\
    msg:'<INCIDENT_ID>: <description>',\
    tag:'application-multi',\
    tag:'attack-TYPE',\
    tag:'paranoia-level/1',\
    tag:'<INCIDENT_ID>',\
    tag:'virtual-patch',\
    severity:'CRITICAL',\
    ver:'<INCIDENT_ID>-vp/1.0'"
```

#### `tests/<RULE_ID>.yaml` — go-ftw Regression Tests

One YAML file per rule. go-ftw v2 format:

```yaml
meta:
  author: "first-responder"
  enabled: true
  name: "<RULE_ID>.yaml"
  description: "Tests for rule <RULE_ID> (<INCIDENT_ID>)"
rule_id: <RULE_ID>
tests:
  - test_title: "<RULE_ID>-1-attack"
    stages:
      - input:
          method: "POST"
          uri: "/"
          headers:
            Host: "localhost"
            Content-Type: "application/json"
          data: '{"payload": "attack_string"}'
        output:
          status: 403

  - test_title: "<RULE_ID>-2-benign"
    stages:
      - input:
          method: "GET"
          uri: "/api/health"
          headers:
            Host: "localhost"
        output:
          status: 200
```

**Key requirements**: `rule_id` at top level (required by go-ftw v2), `Host` header in all inputs (nginx requires it), `test_title` (not `test_id`).

**Now research the vulnerability.** Document findings in `REPORT.md` as you go:

- What is the attack vector? (request method, headers, body format, URI pattern)
- What patterns would appear in HTTP traffic? (specific strings, encoding, protocol markers)
- What does CRS already detect? (check existing rule categories: 93x for injection, 941 for XSS, 942 for SQLi, etc.)
- What might CRS miss? (new protocol patterns, encoding bypasses, content-type blind spots)
- What is the CVSS / severity? Is there active exploitation?
- Advisory links, PoC references, vendor patches

> **References for building detection logic**:
> - Variables to target (`ARGS`, `REQUEST_HEADERS`, etc.) → [variables-and-collections.md](variables-and-collections.md)
> - Operators and transforms (`@rx`, `@pm`, `t:urlDecodeUni`, etc.) → [operators-and-transforms.md](operators-and-transforms.md)
> - Anomaly scoring and paranoia levels → [anomaly-scoring.md](anomaly-scoring.md), [paranoia-levels.md](paranoia-levels.md)

### 2) Start test environment

Start **both** engines so you can test cross-engine. Run them one at a time (they share port 8080).

```bash
# ModSecurity + CRS + Albedo (start here)
docker compose -f assets/docker/docker-compose.yaml up -d

# Coraza + CRS + Albedo (swap to this for cross-engine testing)
docker compose -f assets/docker/docker-compose.yaml down
docker compose -f assets/docker/docker-compose.coraza.yaml up -d
```

The environment is shared across incidents — one WAF instance serves all evaluations.
**Review tuning parameters** (`PARANOIA`, `BLOCKING_PARANOIA`, `ANOMALY_INBOUND`, `ANOMALY_OUTBOUND`) before first use — see `assets/docker/.env.example`.

> **Deep dive**: ModSecurity environment, image tags, env vars → [modsec-crs-testing-reference.md](modsec-crs-testing-reference.md).
> Coraza-specific env vars, paths, and cross-engine comparison → [coraza-testing-reference.md](coraza-testing-reference.md).
> Compose files: `assets/docker/docker-compose.yaml` (ModSecurity), `assets/docker/docker-compose.coraza.yaml` (Coraza).

### 3) Build and run probes (OODA: Orient → Decide)

**CRITICAL: Test each payload across multiple content-types.** The same attack string can be blocked in one content-type and missed in another (e.g., JSON body processor detects prototype pollution but multipart form fields do not).

Convention:
- **Attack probes**: prefix "P" (P1, P2, ...). For each attack, test JSON, multipart, and urlencoded variants.
- **Benign controls**: prefix "C" (C1, C2, ...). Normal requests that must NOT be blocked.

Probes:

```bash
# Attack probe
curl -i -H "Host: localhost" "http://localhost:8080/?payload=attack_string"

# Same payload, different content-types:
curl -i -H "Host: localhost" -H "Content-Type: application/json" \
  --data '{"payload":"attack_string"}' "http://localhost:8080/"
curl -i -H "Host: localhost" -F 'payload=attack_string' "http://localhost:8080/"
curl -i -H "Host: localhost" -H "Content-Type: application/x-www-form-urlencoded" \
  --data 'payload=attack_string' "http://localhost:8080/"
```

### 4) Collect and analyze WAF evidence

Container names differ by engine: `crs-waf` (ModSecurity) vs `crs-coraza` (Coraza).

```bash
# Extract JSON audit entries from container logs
# ModSecurity:
docker logs crs-waf 2>/dev/null | grep '^{' > incidents/<INCIDENT_ID>/audit-modsec.log
# Coraza:
docker logs crs-coraza 2>/dev/null | grep '^{' > incidents/<INCIDENT_ID>/audit-coraza.log

# Analyze (repeat for each engine's log file)
python scripts/analyze_log.py incidents/<INCIDENT_ID>/audit-modsec.log --summary
python scripts/analyze_log.py incidents/<INCIDENT_ID>/audit-modsec.log --top-rules 20
python scripts/analyze_log.py incidents/<INCIDENT_ID>/audit-modsec.log --rule-id 942100 --detail
```

The `--summary` output includes a per-request breakdown showing method, URI, HTTP status, and matched rule IDs for each logged transaction. Record these in the REPORT.md.

> **Deep dive**: Log format details (JSON vs Native), top-talker CLI patterns, error log analysis → [log-analysis-steering.md](log-analysis-steering.md).

### 5) Sandbox comparison

For each critical payload, run CRS Sandbox comparison:

```bash
curl -H "x-format-output: json-matched-rules" \
  "https://sandbox.coreruleset.org/?payload=attack_string"
```

Compare local results with Sandbox. Treat Sandbox as reference signal, not authoritative for your environment.

> **Deep dive**: Sandbox API headers (`x-backend`, `x-crs-paranoia-level`, `x-format-output`), reproducible reporting patterns → [crs-sandbox-reference.md](crs-sandbox-reference.md).

### 6) Produce consolidated summary

Fill in the results table in `incidents/<INCIDENT_ID>/REPORT.md`:

| Engine | Engine Version | CRS Version | Payload | Result | Rule IDs | FP/Bypass | Notes |
|--------|---------------|-------------|---------|--------|----------|-----------|-------|
| ModSecurity | latest | latest | P1 | blocked/missed | 934100 | No | |
| ModSecurity | latest | latest | P1-MP | missed | (none) | Bypass | content-type bypass |
| ModSecurity | latest | latest | C1 | pass | — | No | |
| Coraza | latest | latest | P1 | blocked/missed | 934100 | No | |
| Coraza | latest | latest | P1-MP | missed | (none) | Bypass | content-type bypass |
| Coraza | latest | latest | C1 | pass | — | No | |

Flag any differences between engines — they indicate engine-specific behavior that affects deployment decisions.

### 7) Write virtual-patch rules (OODA: Act)

When probes reveal gaps, write SecRules in `incidents/<INCIDENT_ID>/rules.conf`.

Conventions (see full template in [Step 1](#1-research-the-risk-and-scaffold-workspace)):
- Use the **allocated rule ID range** (printed by `new_incident.sh`)
- Include CVE/incident tag and `virtual-patch` tag for traceability
- `severity:CRITICAL` for confirmed exploit patterns; `severity:WARNING` for indicators
- `deny,status:403` for confirmed exploits; `pass` with anomaly score for indicators
- `ver:'<INCIDENT_ID>-vp/1.0'`

```bash
# Validate syntax
python scripts/validate_rule.py incidents/<INCIDENT_ID>/rules.conf

# Lint regex patterns for ReDoS/performance
python scripts/lint_regex.py incidents/<INCIDENT_ID>/rules.conf -v
```

> - Operators and transforms → [operators-and-transforms.md](operators-and-transforms.md)
> - Regex quality and ReDoS prevention → [regex-steering-guide.md](regex-steering-guide.md)

### 8) Activate rules and re-test

```bash
# Assemble all active incident rules into the WAF's custom rules file:
bash scripts/assemble_rules.sh
# Or assemble only specific incidents:
bash scripts/assemble_rules.sh CVE-2025-55182 RISK-2026-003

# Restart WAF to load new rules — use the compose file for the active engine:
# ModSecurity:
docker compose -f assets/docker/docker-compose.yaml down && \
docker compose -f assets/docker/docker-compose.yaml up -d
# Coraza:
docker compose -f assets/docker/docker-compose.coraza.yaml down && \
docker compose -f assets/docker/docker-compose.coraza.yaml up -d

# Verify rule count increased (container name differs by engine):
docker logs crs-waf 2>&1 | grep "rules loaded"     # ModSecurity
docker logs crs-coraza 2>&1 | grep "rules loaded"   # Coraza
```

Re-run the probes that were previously missed on **both engines**. Verify benign controls still pass.

### 9) Codify as go-ftw regression tests

Write test YAML files in `incidents/<INCIDENT_ID>/tests/` — one file per rule ID. Include both attack payloads (should block) and benign controls (should pass). See the test format in [Step 1](#1-research-the-risk-and-scaffold-workspace) above.

> **Deep dive**: go-ftw test file format, `rule_id` requirements, `--cloud` mode, `.ftw.yaml` config, `testoverride`, and advanced usage → [go-ftw-reference.md](go-ftw-reference.md).
> Cross-engine testing (ModSecurity vs Coraza) → [coraza-testing-reference.md](coraza-testing-reference.md).

```bash
# Run tests for this incident only (on whichever engine is currently up):
go-ftw run --cloud --config assets/docker/.ftw.yaml -d incidents/<INCIDENT_ID>/tests/

# Run tests across ALL incidents (scans tests/ subdirectories):
for d in incidents/*/tests; do
  echo "--- $(dirname "$d") ---"
  go-ftw run --cloud --config assets/docker/.ftw.yaml -d "$d" || true
done

# Cross-engine regression: swap engine and rerun
# 1. Stop current engine, start the other:
docker compose -f assets/docker/docker-compose.yaml down
docker compose -f assets/docker/docker-compose.coraza.yaml up -d
# 2. Rerun the same tests:
go-ftw run --cloud --config assets/docker/.ftw.yaml -d incidents/<INCIDENT_ID>/tests/
```

### 10) Manage rule lifecycle

```bash
# List all incidents and their rule counts:
bash scripts/assemble_rules.sh --list

# Retire an incident (remove its rules from the active set):
# Either delete/empty its rules.conf, then re-assemble:
bash scripts/assemble_rules.sh

# Clear all custom rules:
bash scripts/assemble_rules.sh --clear
```

When upstream CRS adds coverage for an incident, retire the virtual-patch by emptying the rules.conf but keep the tests as regression coverage.

## Documentation Template

Each incident workspace includes a `REPORT.md` with this structure:

- Risk ID / advisory link
- Hypothesis summary
- Probe matrix (with content-type variants)
- Results: CRS baseline (no custom rules)
- Results: with virtual patches
- Sandbox comparison
- Gaps identified
- Custom rules written (with rule IDs)
- Follow-up actions (regression tests, tuning, owner, due date)
- Environment details (engine, CRS version, PL, threshold)

## Guardrails

- Do not deploy global disables as emergency workaround.
- Prefer scoped controls by URI/method/variable.
- Keep first-response rules temporary until validated and reviewed.
- Always include benign controls to detect overblocking.
- Test across content-types — content-type bypass is a common WAF evasion.
- Use `assemble_rules.sh` to compose rules — never edit `custom-rules.conf` directly.
- Rule ID ranges are per-incident — do not reuse IDs across incidents.
- Place go-ftw test files in `incidents/<ID>/tests/` only — do not put `.ftw.yaml` or non-test YAML in that directory.

## Best Practices

- **Research first, build second** — scaffold the incident workspace during research, not after rule writing.
- Document every hypothesis, probe, and finding in `REPORT.md` — even negative results are valuable.
- Always include both attack and benign probes to detect overblocking early.
- Test across content-types (form, JSON, multipart) — content-type bypass is a common WAF evasion.
- Keep first-response rules temporary with clear `ver` tags and review dates.
- Run regression tests per-incident (`go-ftw -d incidents/<ID>/tests/`) before assembling into production.

## What to Avoid

- Deploying global disables or broad exclusions as emergency workarounds.
- Skipping benign (negative) test cases — leads to overblocking in production.
- Reusing rule IDs across different incidents — causes conflicts during assembly.
- Editing `custom-rules.conf` directly instead of using `assemble_rules.sh`.
- Filing a virtual patch without environment details (engine version, CRS version, PL, threshold).
